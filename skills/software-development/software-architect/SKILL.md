# Software Architect

## Overview

Use this skill when a task requires architecture-level thinking: component boundaries, data flow, integration patterns, scalability, maintainability, and trade-offs.

## When to Use

Use when you need to:

- design a new system or subsystem
- choose between architecture patterns
- define service/module boundaries
- review scalability or reliability concerns
- plan integration across multiple components
- align technical design with business constraints

## Working Style

1. Clarify goals, constraints, and non-goals.
2. Map the system into responsibilities and boundaries.
3. Identify data flow, dependencies, and failure modes.
4. Compare options and trade-offs explicitly.
5. Choose the simplest design that meets requirements.

## Core Concerns

- Separation of concerns
- Coupling and cohesion
- Scalability and performance
- Reliability and operability
- Security and trust boundaries
- Observability and maintainability

## Common Pitfalls

- Overdesigning before understanding the problem.
- Ignoring operational concerns.
- Mixing implementation details with architecture decisions.
- Choosing patterns because they are trendy rather than necessary.

## Verification Checklist

- [ ] Goals and constraints are clear
- [ ] Boundaries are well defined
- [ ] Trade-offs are documented
- [ ] Failure modes were considered
- [ ] The design is simple enough to maintain
