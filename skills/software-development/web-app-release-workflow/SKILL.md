# Web App Release Workflow

Use this skill when modifying a full-stack web app repo that ships via Docker Compose, especially when the change touches frontend presentation, auth/login, static assets, tests, or production deploys.

## Common workflow
1. Inspect the current implementation and file layout.
2. Identify the minimum source changes needed.
3. Patch source files.
4. Run the smallest useful verification first, then expand:
   - frontend build
   - API/unit tests
   - e2e tests
   - deploy script / health check
5. If deploy is involved, verify the runtime env and container

## Frontend media changes
- For KSR888 mobile/provider-image cleanup and legacy-image removal, see `references/ksr888-mobile-provider-image-cleanup.md`.
- Put source media under `web/public/<folder>/...` so Vite serves it at `/folder/...`.
- Reference public assets with root-relative URLs like `/video-website/file.mp4` or `/images-website/file.jpg`.
- When the user says they added new videos/images in a public media folder, inspect the actual folder contents and update the source asset array to those files. Do not merely rename visible labels to `Video 1`, `Video 2`, or similar generic placeholders unless explicitly requested.
- Avoid cluttering video cards with generic visible labels such as `Video 1-6` or `Koleksi Baru 01-06`; if filenames are opaque UUIDs and no user-facing titles are provided, prefer clean video presentation, subtle `Reel 01` metadata, or no title overlay.
- For hero/video or hero/image sliders:
  - keep the media wrapper `relative` if you use absolute overlays
  - use `video` with `autoPlay muted loop playsInline preload="metadata"`
  - keep the current index in state and clean up timers in `useEffect`
  - provide manual prev/next controls and indicator dots if the slider is user-facing
  - when showing both a video and a still-image preview, keep each carousel state independent so one does not restart the other
- When the user says they added/replaced “video 1-6” in a public media folder, inspect `web/public/<folder>` directly, update the source array/order/labels (not `web/dist`), and verify every production asset URL with HTTP 200/HEAD after deploy
- When the user asks to remove one homepage reel, delete the source array entry in the landing component, rebuild, and grep the built bundle for the stale visible label or filename before declaring it gone.
- When the user asks to add all videos from a folder, copy every file into a Vite-served public folder, switch the UI from a single preview to a grid/showreel if needed, and use neutral per-item labels like `Reel 01` when filenames are ugly; see `references/bulk-video-asset-ingestion.md`

- When a user provides a screenshot/mockup for a checkout page, analyze the reference image first and match structure before micro-styling.
- Prioritize spacing, border weight, radius, and field grouping over copy changes unless the user explicitly asks for new text.
- For static checkout pages maintained in `web/public/`, treat that file as source of truth and sync mirrors/build artifacts only after the source is correct.
- Rebuild, deploy, then verify the live page with HTTP or screenshot-based checks; bundle grep alone is not enough for visual changes.
- When the user asks for *very precise* matching, do a compact mobile-first pass: topbar height, section gaps, field height, border radius, and button size first; only then polish font weight and copy density.

## Mobile browser UX
- When the user says the mobile site looks `berantakan`, treat it as a content-density problem first: remove duplicate hero/title blocks, collapse repeated CTA rows, and shorten the intro before changing deeper layout CSS.
- For cluttered mobile homepages, clean the wrapper content first and preserve the navigation/menu structure; use compact helper copy plus 2–3 primary CTAs instead of stacking repeated marketing blocks.
- See `references/mobile-homepage-cleanup-pattern.md` for the exact duplicate-hero / duplicate-CTA cleanup pattern from this session.
- When the user asks to make the site easier to use on phones, prioritize mobile-first source edits in `web/src/components/*` and `web/src/styles/index.css`.
- For Android/iOS readiness, ship the web app as a PWA first:
  - add/update `web/public/manifest.webmanifest`
  - provide at least one SVG icon plus a maskable variant
  - link manifest, favicon, and apple-touch-icon in `web/index.html`
  - verify the web build still passes before adding native wrappers
- If the app is going to be wrapped for Android/iOS with Capacitor, install the native packages after the PWA shell is stable and keep the native wrapper as a second step, not the first.
- When making the app easier for first-time users, add a visible `Panduan` / getting-started surface that explains the shortest path to value in 1-2-3 steps, and pair every advanced feature with a simple example or sample input.
- Default mobile fixes that usually help:
  - collapse dense top navs into stacked rows or scrollable chips
  - shorten CTA text on small screens when labels wrap or clip
  - stack primary/secondary buttons vertically if a two-column row feels cramped
  - hide or soften heavy background video/media on phones if it hurts readability or performance
  - keep article/news cards single-column on phones and reduce image height
  - add `overflow-x: hidden`, `touch-action: manipulation`, and small `scroll-padding-top` for anchor usability
- For news portals that should feel like a majalah/portal berita, keep the mobile hierarchy editorial and low-clutter:
  - masthead first, then one dominant cover story
  - follow with a compact breaking line / editor’s pick rail
  - keep category chips horizontally scrollable rather than wrapping into a tall block
  - split long portal headlines with explicit line breaks if they clip on narrow screens
  - keep one clear CTA row and prefer stacked buttons on very narrow widths
- Use touch-target hygiene on small screens: favor full-width buttons, avoid crowded chip rows, and keep small labels visually separated so they are easy to tap.
- Verify the result with a real mobile viewport screenshot (for example Chromium headless at `390x844`) and inspect for clipped text, wrapped buttons, cramped chips, or horizontal overflow before calling it done.
- See `references/mobile-news-portal-layout.md` for the mobile news-portal pattern and verification checklist.
- See `references/mobile-pwa-capacitor.md` for the PWA + Capacitor Android/iOS readiness pattern and verification checklist.
- See `references/mobile-pwa-capacitor-and-wa-crm-voice.md` for the session notes on turning the web app into a mobile-ready PWA, adding the WA CRM voice-agent parse layer, and using a `Panduan` quick-start surface.

## Production UX evaluation
- When the user asks to evaluate/audit/review a live app, treat it as an evidence-backed report request, not an implicit deploy request. Do live HTTP/API/container/browser checks, run build/typecheck when source is available, and separate findings into P0/P1/P2 priorities.
- If the user supplies a numeric approval threshold, use it as the go/no-go gate before making changes. Do not start fixes unless the audit clears the stated score.
- Guardrail: do not start or continue source edits during a pure evaluation unless the user explicitly asks to fix/update. If partial edits exist from a previous request, mention them separately and avoid deploying them as part of the audit.
- Include mobile and desktop smoke checks: title/H1, visible CTA labels, console errors, horizontal overflow, DOM/media counts, and scroll height. If Playwright's bundled browser is missing, use an installed browser executable such as `/snap/bin/chromium` with `--no-sandbox`.
- During public-site audits, explicitly check `index.html` cache headers, duplicate static media, visible responsive-label duplication, customer-facing source/model/provider labels, and generated data/disk growth. See `references/production-ux-evaluation-checklist.md`.
- For subjective go/no-go decisions, run a 3-role panel: critic, advocate, and voter. Score each out of 100, compute average and median, and prefer approval only when the final score is above 60 and there is no critical blocker. If the panel disagrees, report the disagreement instead of collapsing it into a single opaque judgment. See `references/score-gated-release-review.md`.

## Dashboard and studio simplification
- When the user asks to make the app easier to use, treat it as a progressive-disclosure task, not a feature-removal task.
- See `references/content-template-and-evaluation-panels.md` for the pattern of adding professional content-template and content-evaluation surfaces directly into the dashboard.
- Make the default path obvious: one primary CTA, one prompt field, and short outcome-first labels.
- Prefer direct feature chips/cards over generic `Quick Create` blocks if the goal is clarity.
- When adding a new dashboard feature, wire it through all four surfaces together: quick card, section union/state, panel import/render, and backend route/db if the feature persists data.
- For service-registration features (`Jasa`), restrict the public catalog to legal services, require an explicit legal confirmation checkbox, and save stable fields such as business name, service name, category, mode, city, contact, description, and optional website.
- Remove decorative controls that do not help the first task (for example a search box that does not filter anything meaningful).
- Rename technical labels to plain language unless the user explicitly wants expert terminology.
- Keep advanced controls available, but move them below a visible `advanced`/`pengaturan lanjut` boundary.
- See `references/dashboard-feature-injection-and-service-registration.md` for the dashboard-card + legal service registration pattern and the build verification checklist.
- If the user asks to remove manual model choices across features, remove the selectors from the client UI, keep a fixed automatic server-side default, and replace visible copy with neutral text like `Otomatis` / `Pengaturan otomatis` rather than `Model`. Update both the chat path and the non-chat studio path so no feature still exposes a model dropdown by accident.
- When selectors are removed, also stop sending client-chosen model ids in request payloads for the affected feature; the backend should infer the model from the feature/catalog default instead. Preserve internal pricing/ledger logic, but hide the choice from the user.
- Verify with a production build after the copy/layout pass, then do a quick browser smoke check to confirm no visible `select`/`Model` control remains in the rendered feature panels.

## Landing studio cards and visible model cleanup
- When the user asks for a UI-only color cleanup (for example all buttons should become transparent-white and no random colors), change only the requested visual property. Do not alter layout, spacing, content, cards, badges, or feature structure. For app-wide button-only color normalization, prefer a scoped global CSS override for `button`/CTA links rather than touching every component class; see `references/button-color-uniformity.md`.
- For authenticated dashboard cleanup, Quick Create cards should navigate to their matching feature section instead of being decorative. If the user asks to neutralize colored dashboard panels, use dashboard-scoped transparent/neutral panel overrides and preserve card/grid layout; see `references/dashboard-quick-create-routing-and-neutral-panels.md`.
- When the user says the studio/landing looks ugly because of visible models, interpret it as a customer-facing UI cleanup unless they explicitly ask to remove backend model catalog/provider support.
- Prefer feature/benefit chips over provider/model names on landing cards. Examples: `Text to Video`, `Keyframe`, `Reference Media`, `Native Audio`, `Voice Agent`, `Help Desk`, `Custom Agent`.
- For promo campaigns on the home page, treat the landing CTA and the checkout amount as one coupled change: update hero copy, price cards, and backend payable amount together, ideally via an env-driven promo percentage so future campaign edits stay consistent.
- Distinguish public landing cleanup from logged-in dashboard UX. If the user says “hanya tampilkan fitur saja model tidak perlu,” hide model/provider names on the public landing cards first; do not remove model visibility from the authenticated Studio unless they explicitly ask for the dashboard too.
- If the user later asks to keep models available after login, restore/keep the dashboard Studio model dropdown (for example `Model Tersedia`) and result provider/model labels while keeping the public landing feature-only.
- When the user wants a feature noticed by users (for example Voice Agent), move that feature card to the front of the landing studio grid and add a direct CTA.
- After build, grep the production bundle for unwanted customer-visible model/provider strings and for the new feature/CTA copy.

## Brand/logo cards
- For model/provider badges, prefer official icon sets before inventing generic placeholder logos.
- `react-icons/si` often has good coverage for brand marks; map each card to a `brand` token and render the icon through a reusable component.
- When the user asks for official/resmi logos, avoid initials or generic placeholders. If `react-icons/si` lacks the exact mark, save a provider SVG/favicon under `web/public/brand-logos/` and add `logoSrc`/`logoAlt` fields to the card data.
- Probe icon exports before importing them; installed `react-icons/si` may lag Simple Icons (for example, MiniMax exists as a Simple Icons slug but not as `SiMinimax` in some package versions).
- Keep the card text, icon/image, and background treatment in data rather than hard-coded per tile; that makes future brand swaps a one-line change.
- After deploy, verify new public logo URLs directly with `curl`/HTTP 200 checks in addition to the app health endpoint.

- For customer-facing footer/header chrome, avoid exposing operational implementation status such as `API online/offline` and raw app version strings (`v1.0.0`) unless explicitly requested for an admin/debug surface. If removing it, check both authenticated dashboard header badges and public landing footer text, then grep the built JS bundle for the old fragments (`API online`, `Full VIP Experience • API`, and version fragments like ` • v`).
- When a user asks to deploy or replace a landing page from a provided HTML file, inspect the file first for third-party branding or impersonation risk. If the file mirrors another brand, rewrite it into an original brand-safe page before deploy instead of copying the branding verbatim. Keep the original visual style only if it can be cleanly rebranded.

## Auth / Google Sign-In changes
- Prefer Google Identity Services button rendering over custom click handlers that only call `prompt()`.
- When GIS is used, make sure CSP allows the Google endpoints needed by the script and iframe flow.
- If the browser reports failures around `https://accounts.google.com/gsi/transform`, inspect headers before rewriting auth logic: Helmet's default `Cross-Origin-Opener-Policy: same-origin` can break GIS popup/iframe flows. Prefer `same-origin-allow-popups`, allow Google/GStatic in CSP `frameSrc`/`scriptSrc`/`connectSrc`/`imgSrc`, then redeploy and verify headers with `curl -I`.
- Ensure both backend and frontend env vars are present when the app needs Google login:
  - `GOOGLE_CLIENT_ID`
  - `VITE_GOOGLE_CLIENT_ID`
- If e2e tests stub Google auth, stub the same method the UI actually uses (`renderButton` vs `prompt`).
- When switching from demo/Google-only mode to live production, remove the demo-only `410 Gone` handlers from backend login/register routes, expose the matching email/password UI, and update API/e2e stubs from rejection responses to successful session responses.
- If the app uses seeded SQLite users, make live smoke tests delete their temporary DB/WAL/SHM files before seeding so duplicate-user state does not make the test flaky.

## Playwright / local web server pitfalls
- If a test config shells out to `cross-env`, use the local binary path when the environment does not expose global binaries:
  - `./node_modules/.bin/cross-env`
- If a selector-based test starts failing after a UI change, align the stub or selector with the real user-visible flow instead of forcing the old text/button path.
- Scope selectors when labels are duplicated across tabs and submit buttons, for example `page.locator('form').getByRole('button', { name: 'Login' })` instead of a page-wide `getByRole`.
- If tests keep seeing old API behavior after a source fix, check for stale Docker containers or proxies bound to the backend port before retrying the same test.

## Live feature/API integrations
- When the user asks to make dashboard/sidebar features “real live,” convert inert links/cards into active panels and pair each panel with an auth-protected backend route; do not leave UI-only/demo behavior.
- For WhatsApp CRM voice-agent work, keep the first pass as a deterministic parse-and-route layer:
  - expose a dedicated `/api/wa-crm/voice/parse` endpoint
  - parse transcript intent into a small fixed set (`add_contact`, `create_follow_up`, `draft_reply`, `summarize_conversation`, `log_note`)
  - extract phone/name/company/due-time hints with lightweight heuristics before adding heavier speech AI
  - mount the panel in the dashboard as its own section so it complements CRM instead of hiding inside generic chat
- For live user-to-user chat in this app, prefer a small authenticated REST + Server-Sent Events implementation before introducing heavier WebSockets: persist messages in SQLite, use REST for rooms/history/send, use `EventSource` with a query-token stream for realtime fanout, keep AI Chat and Chat Live as separate dashboard sections, and verify with API tests plus mobile Playwright. See `references/live-user-chat-sse.md`.
- For admin dashboards that expose editable file-backed content (for example article drafts), add a split list/editor view, keep the edit payload whitelist-only, make the selected item immutable by id/slug, and persist through a PATCH route that rewrites the JSON, HTML, and index files together. Use a ref to protect the selected item from polling refreshes and verify with both API tests and `npm run build:web`. See `references/admin-article-draft-editor.md`.
- When the user asks to remove a dashboard feature, do a full decommission: remove the navigation/card, the React section branch, the backend router mount, any feature-specific static asset route, and then update tests to assert 404 or absence. Also search for persistence/schema leftovers (tables, enums, ledger reasons, cron jobs, worker scripts) and decide explicitly whether they need a migration/drop path or can stay as dormant data. See `references/feature-decommissioning-and-route-removal.md`.

## Admin dashboard releases
- When the user asks for an admin dashboard at a path like `/admin`, add both a frontend SPA route and an auth-protected backend summary endpoint; do not ship UI-only admin screens.

- Admin users in this repo are governed by `ADMIN_EMAILS`; existing users may retain an old role until login/role refresh or DB update, so verify both no-auth `401` and non-admin `403` paths before declaring access ready.
- Useful admin cards can be derived from existing SQLite tables: users, generation history, payment orders, credit accounts, and visitor analytics events. Avoid exposing secrets or sensitive raw env values in the admin UI or verification output.
- For visitor counts, add lightweight server-side tracking before SPA fallback: create a `visitor_events` table, set an HTTP-only long-lived visitor cookie, hash IP/user-agent-related data if needed, and make analytics best-effort so it never blocks public page loads.

This repo's `credit_ledger` schema uses `amount`, `balance_after`, `reference_type`, `description`, and `metadata`, not legacy `type`/`metadata_json` fields.

Use deterministic smoke email patterns (`*@example.test`, `smoke-%@example.com`, etc.), clean them up after verification when safe, and make admin `Total User Real` exclude known smoke/test patterns while optionally exposing a raw `totalRegisteredUsers` for audit.
- If register or Google Sign-In creates the user but then returns `500` / `SQLITE_CONSTRAINT_UNIQUE` and fails to redirect to dashboard, trace post-user hooks like signup credit grants before changing the frontend. In this repo a stale global ledger reference (`signup_bonus`) caused every new signup bonus to collide on `UNIQUE(reference_id, reason)`. Make credit grants idempotent: insert the ledger row with `INSERT OR IGNORE` first, only update balances when the ledger insert changes a row, and use per-user references such as `signup_bonus:<userId>`.

Parse `resultJson` defensively and show credit cost, provider, model, and a lightweight status badge instead of raw JSON.

## Deploy workflow
- Read production env from `.env.production` first; `.env` may also be loaded if present.
- Build and start through the project script, not by hand-editing containers.
- If the deployment directory is not a git repository, do not block on `git status`; verify readiness from `package.json`, deploy scripts, env files, and current Docker Compose state instead.
- When the user says to deploy “the previous update” / “pembaruan yang sebelumnya,” treat it as a request to redeploy the current project state unless they specify a different revision; do the build/deploy/verification flow immediately rather than asking which commit.
- After deploy, verify the health endpoint and container status.
- For imported PHP hosts, also restart/recreate the live web service after deploy so the new PHP code, DB-backed images, and cache-cleared state appear immediately.
- A successful build is not enough; re-probe the public URL after the restart.
- See `references/ksr888-gamexaglobal-single-api-and-sync.md` for the KSR888 single-API GameXaGlobal source-of-truth + provider-image persistence note.
- See `references/ksr888-game-launch-troubleshooting.md` for the documented launch payload fields, returned URL preservation rules, and Docker/PHPUnit verification steps.
- See `references/ksr888-gamexaglobal-dedup-and-admin-transactions.md` for the GameXaGlobal duplicate provider/game cleanup, legacy schema gotchas, `/transactions-overview` live transaction/turnover wiring, and the `/transactions` primary admin alias pattern.
- For host-specific landing pages on a subdomain, verify the public HTTPS host directly after deploy; a healthy app container is not enough.
- If the app fails to start because the host port is already in use, move the host-side mapping via env/compose instead of changing the container port.
- A successful backend health endpoint does not prove the subdomain is live.
- The app can be healthy while Caddy is still serving stale config or stale TLS state.
- Always verify the public HTTPS host directly after restarting the proxy.
- If the public HTTPS host returns 403/1010 but the local origin is healthy, compare responses with and without a browser-like user-agent before touching app code; that usually points to Cloudflare/WAF/Bot rules, not a backend regression.
- For public-site performance, prefer cache headers/CDN/edge caching over rotate-proxy ideas; if the site feels slow, first inspect asset caching and origin headers. See `references/public-cache-and-proxy.md`.
- If browser navigation/snapshot tools hang, run a small Playwright smoke probe from the terminal against the public URL: launch Chromium headless, wait for a stable landing selector and login button, capture page/console errors, and fail on any error.
- If the site looks blank or broken in the browser, verify the live page with a headless Chromium screenshot before changing source; build artifacts can be fine even when the deployed runtime is stale or misconfigured.
- If browser smoke helpers fail with Chromium singleton/profile errors (`Failed to create socket directory`, `ProcessSingleton`, `DevToolsActivePort`), do not keep retrying the same launch. Clear stale `/tmp/org.chromium.Chromium.*` directories if needed, then fall back to HTTP verification: fetch the live HTML with a browser-like UA, resolve the bundle, and grep for the feature copy you expect. For SPA routes like `/admin`, the HTML title can still reflect the shell route, so verify the hydrated bundle or the live DOM copy rather than the title alone. If even an isolated `HOME`/`XDG_CONFIG_HOME`/`--user-data-dir` launch still aborts with `ProcessSingleton`, treat browser automation as blocked in this environment and stop retrying identical Chromium launches.
- For imported PHP hosts like KSR888, use the same fallback mindset: if Chromium/Playwright hits snap singleton/profile/socket-directory errors, switch to browserless HTTP or Jest smoke verification for the live route instead of burning time on repeated browser launches.
- When live verification only needs to prove catalog/navigation behavior, prefer exact `curl` probes against the public HTTPS route and grep for route labels/links instead of waiting on full browser automation. This is especially useful after cache-only or controller-only performance fixes.
- On this host, if the only available Chromium is a snap wrapper and Playwright launches fail with `ProcessSingleton` / socket-directory errors, switch the e2e gate to a Jest or HTTP smoke test that covers the same route/API contract instead of forcing browser automation. Record the limitation in a reference note so later sessions do not burn time on the same browser path.
- For imported PHP admin/backoffice login flows, verify whether the host expects a separate PIN field in addition to username/password. If so, update the form, handler, and schema together, seed the admin row in the live DB, and verify with a container-local `php -r` probe from the login script directory before trusting the public HTTPS path. See `references/ksr888-admin-login-pin.md`.
- On KSR888 specifically, the admin login surface is `/support` (with `/admin` and `/admins` aliases), the handler accepts either username or email in the same field, and successful admin auth should land on `/dashboard`. Keep this behavior documented in `references/ksr888-admin-login-email-or-name.md`.
- If admin website settings seem to disappear, check whether `/setting` is trapped inside a developer-only middleware branch instead of the admin branch; restore the sidebar link and route exposure for admin users, then verify the page returns `200`. See `references/ksr888-admin-setting-visibility.md`.
- See `references/browser-verification-fallback.md` for the exact terminal fallback recipe and SPA-route caveat.
- For PHP-hosted assets behind the same proxy/CDN, verify the exact static file you changed with a cache-busting query (`?v=<timestamp>` or similar) plus `Cache-Control: no-cache`; a plain asset URL can appear stale even after the container already has the new file. If a visible change still does not appear, consider that the browser/edge may be pinning the old filename; a safe workaround is to ship a renamed stylesheet copy (for example `*.luxury.css`) and point the page at that new filename with a fresh version string. When changing KSR888 desktop/mobile theming, the source-of-truth CSS is `KSR888/site/dekstop/Content/Theme/dekstop.css` / `KSR888/site/mobile/Content/Theme/mobile.css`, and the fastest proof is to grep the live stylesheet response for your new markers before judging the homepage visually. If a site-wide media filter makes banners look desaturated, add a late banner-specific override (`.banner img`, `.banner-carousel img`, `.banner-carousel .item img`, `.banner-carousel .carousel-inner img`) to keep the banners colorful. If the page still looks unchanged after a CSS deploy, inspect the PHP entry page and bump the stylesheet query string there too; do not assume the container build alone is enough. For imported PHP hosts like KSR888, add a host-specific note/reference when the fix depends on live DB env sync, stale browser-profile cleanup, or banner-inspired category-strip CSS so the next deploy starts from the known pitfalls.

- For Vite/React SPAs, raw homepage HTML usually only proves the shell loaded; verify newly added UI copy by grepping the built/live JS bundle or running a Playwright DOM smoke check after hydration.
- If a deploy e2e stub/catalog omits optional numeric fields, normalize them before display instead of calling methods like `.toLocaleString()` on possibly missing values; this prevents unrelated UI crashes from blocking release verification.
- Do not store manual backups that include copied `tests/` directories inside the project root when Jest scans broad test patterns; move backups outside the repo/project tree or exclude test files so deploy/predeploy does not run stale copied tests.
- If Playwright times out during form filling even though the locator resolves, inspect the failure context and consider increasing the project test timeout (for example `timeout: 60000`) after confirming the UI flow is valid; do not repeat the same deploy command unchanged.

## Pitfalls
- Static files in `web/dist/` are build artifacts; do not treat them as source.
- Absolute overlays inside video cards break if the parent is not `relative`.
- Google Sign-In can appear “broken” when the real issue is CSP or OAuth origin mismatch.
- Build/test success does not guarantee production success unless the runtime env vars are present.
- For provider launch flows that depend on an external catalog, verify the remote lookup path with a missing-local-row case before declaring the game unavailable.
- TypeScript may fail Docker `npm run build` on unrelated strictness errors that local frontend-only builds miss; fix small backend typing regressions (for example recursive functions needing explicit return types) before redeploying.
- If the user pastes real API keys, treat them as secrets immediately: write them only to env storage, mask all verification output, and recommend key rotation if they appeared in chat/tool logs.
